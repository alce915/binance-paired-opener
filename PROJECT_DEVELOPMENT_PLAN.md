# 当前项目后续开发计划

更新日期: 2026-03-27

## 1. 计划目标

本计划基于当前仓库已有能力制定，目标不是推翻重做，而是在现有 `paired_opener` 开单服务、账户监控服务、CLI、Web 控制台、SQLite 持久化和测试基础上，逐步把项目推进到更稳定、更安全、更易维护的状态。

后续开发优先围绕以下四个方向展开:

- 提升真实交易场景下的稳定性与风控能力
- 补齐监控、告警、审计和故障恢复链路
- 优化多账户配置、控制台交互和运维体验
- 建立更完整的测试、文档和发布流程

## 2. 当前基础梳理

从当前代码结构来看，项目已经具备以下基础能力:

- 开单服务: 支持双向开仓、双向平仓、单向开仓、单向平仓、会话暂停、恢复、终止
- API 能力: 已提供会话管理、行情连接控制、模拟执行、白名单读取、账户列表与账户切换等接口
- 监控服务: 已提供账户快照查询和 SSE 推送
- 数据层: 已使用 SQLite 保存会话、轮次和事件，具备基本审计留痕能力
- 运行入口: 已提供 FastAPI 服务入口、CLI 命令和重启脚本
- 测试基础: 已覆盖 engine、gateway、service、config、market stream、账户监控等核心模块

当前阶段更像是“可用的第一版”，后续重点应放在生产可用性，而不是继续单点堆功能。

## 3. 开发原则

- 先稳后快: 优先解决异常处理、风控、恢复能力，再扩展策略功能
- 小步迭代: 每个阶段都要有可以独立交付和回归验证的成果
- 以审计为核心: 所有关键交易动作都要能追溯原因、参数、执行结果和异常
- 面向运维: 设计时同步考虑配置、部署、回滚、日志和排障成本

## 4. 分阶段路线图

### 第一阶段: 稳定性和风控补强

目标: 把现有开单能力从“功能可跑”提升到“异常可控、错误可见、执行可回滚”。

重点任务:

- 补充下单前校验:
  - 增加账户状态、持仓状态、挂单冲突、最小下单金额、最大可承受仓位等统一校验
  - 将校验结果结构化返回，便于 API 和前端统一展示
- 强化异常分类:
  - 区分 Binance 限流、权限错误、网络超时、撮合失败、余额不足、参数非法等错误类型
  - 为不同异常定义重试、终止、人工介入三类处理策略
- 完善会话恢复:
  - 服务重启后不仅标记未完成会话异常，还应支持基于订单状态做恢复判断
  - 明确“可恢复”“不可恢复”“需人工确认”三种状态
- 增强审计数据:
  - 为关键事件增加请求参数快照、交易所返回值摘要、耗时、重试次数、最终结果
  - 增加会话级统计字段，便于后续报表和问题复盘
- 补齐测试:
  - 增加异常分支、恢复分支、边界金额、账户切换、单向开平仓的回归测试

阶段产出:

- 一套更完整的风控校验链路
- 一套可读性更强的错误码或错误分类
- 可验证的会话恢复策略说明和测试用例

### 第二阶段: 监控与告警体系建设

目标: 让系统具备“出了问题能及时发现，发现后能快速定位”的能力。

重点任务:

- 扩展监控维度:
  - 增加服务运行状态、市场连接状态、最近一次下单结果、最近异常、刷新延迟等指标
  - 将开单服务和账户监控服务的状态统一到一个面板中
- 增加主动告警:
  - 支持余额不足、接口异常、监控刷新失败、会话执行异常、长时间未完成等告警条件
  - 预留企业微信、Telegram、邮件或 Webhook 通知接口
- 优化 SSE 事件模型:
  - 规范事件类型、事件字段和前端消费方式
  - 增加心跳、断线重连提示和订阅过滤能力
- 增加操作日志视图:
  - 在控制台中查看最近会话、最近事件、异常摘要、账户状态变化

阶段产出:

- 统一监控视图
- 基础告警能力
- 更清晰的事件流与问题定位入口

### 第三阶段: 多账户与控制台体验升级

目标: 降低人工操作成本，提升多账户场景下的可用性和安全性。

重点任务:

- 完善多账户模型:
  - 明确主账户、分组账户、独立交易账户等角色边界
  - 支持按账户查看会话、按账户过滤事件、按账户限制可交易标的
- 优化账户配置:
  - 统一 `.env`、`config/*.json`、活动账户持久化文件的配置逻辑
  - 增加配置校验、配置示例和启动自检
- 提升控制台体验:
  - 在开单页面增加表单校验、确认弹窗、风险提示、执行进度展示
  - 在监控页面增加账户筛选、排序、异常高亮和数据刷新提示
- 增加只读模式:
  - 支持监控账号和操作账号分离，避免误操作

阶段产出:

- 更清晰的多账户使用流程
- 更稳健的配置管理方式
- 更适合日常值守的前端控制台

### 第四阶段: 工程化与发布能力完善

目标: 让项目进入“可长期维护、可持续交付”的状态。

重点任务:

- 规范文档:
  - 完善安装、配置、启动、排障、接口说明、数据结构说明
  - 补充真实业务流程图和常见操作手册
- 完善测试体系:
  - 补充 API 测试、前端交互关键路径测试、配置兼容测试
  - 引入更明确的测试分层: 单元测试、集成测试、模拟交易测试
- 建立发布流程:
  - 明确版本号规则、发布清单、回滚步骤、数据库兼容策略
  - 增加启动前自检和发布后健康检查
- 提升运维能力:
  - 统一日志格式
  - 增加日志轮转、归档和关键字段检索约定
  - 视情况补充 Docker 或 Windows 服务化部署方案

阶段产出:

- 可交接的项目文档
- 更稳的回归和发布流程
- 面向长期运行的运维规范

## 5. 建议优先级

建议按以下顺序推进:

1. 第一阶段全部内容
2. 第二阶段中的统一监控视图和异常告警
3. 第三阶段中的多账户配置和控制台风险提示
4. 第四阶段中的文档、测试分层和发布流程

如果开发资源有限，优先做“会影响真实下单安全和故障恢复”的内容，延后视觉层和非核心交互优化。

## 6. 里程碑建议

### M1: 可控交易基线

- 完成核心风控校验
- 完成异常分类与错误展示
- 完成服务重启后的会话恢复策略
- 补齐关键回归测试

验收标准:

- 常见异常有明确处理路径
- 关键交易流程具备可追踪审计记录
- 重启后不会出现不可见的悬空会话

### M2: 可观测运行基线

- 完成统一监控面板
- 完成基础告警能力
- 完成事件流规范化

验收标准:

- 值守人员可快速判断系统是否正常
- 关键异常可以在分钟级被感知

### M3: 可维护交付基线

- 完成配置规范化
- 完成主要文档
- 完成发布和回滚清单

验收标准:

- 新环境部署成本明显降低
- 日常维护不依赖单人经验

## 7. 风险与注意事项

- 真实交易相关改动必须先补测试，再开放到生产环境
- 多账户和账户切换逻辑需要重点防止串账户操作
- 风控规则一旦收紧，可能影响现有使用习惯，需要提前评估兼容性
- 监控和告警功能不能只做展示，必须能够支撑排障闭环
- 文档中的编码、环境变量、配置文件说明要统一，避免出现“代码能跑但说明不可用”的情况

## 8. 最终目标

项目后续不应只停留在“能开单”的工具层，而应逐步演进为:

- 可控: 关键交易行为有风控和确认机制
- 可观测: 服务状态、账户状态、异常状态可实时感知
- 可恢复: 异常和重启后有明确恢复路径
- 可维护: 配置、测试、发布、排障都具备标准流程

达到以上目标后，再考虑更复杂的策略编排、自动化执行和更深入的数据分析能力，会更稳妥。
